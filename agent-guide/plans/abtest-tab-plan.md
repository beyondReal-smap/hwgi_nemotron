# 작업 계획서: A/B 테스트 탭 신설

> 작성일: 2026-05-23 | 상태: 초안 (대표님 합의 대기) | 모드: Single (MVP 한 사이클)

## TL;DR

- **무엇**: 좌우 분할 UI에서 두 안(A/B)을 입력하면 동일 페르소나 모집단의 반응을 비교하고, 당사 정보 기반 장단점 + FP 판매전략을 한 화면에서 보여주는 신규 탭
- **왜**: 기존 `/api/analyze`는 단일 안 분석에 특화 — 상품 컨셉·카피 등 두 안 중 어떤 안이 더 통할지 의사결정할 도구가 없음
- **범위**: 신설 8개 + 수정 4개 (api routes 1, services 1, prompts 2, web page 1, components 3 / SiteHeader 1, lib/api.ts 1, main.py 1, schemas.py 1)
- **핵심 리스크**: LLM 호출 횟수(A/B 분석 + 비교 리포트 2회 = 단일 대비 약 1.2배), 당사 정보 텍스트 길이 폭주 시 토큰 초과
- **MVP 완결 기준**: A/B 두 안 입력 → 30~60초 내 양쪽 결과 + 비교 표 + FP 전략 리포트 1화면 표시

---

## 확정 사양 (대표님 합의 사항)

1. **목적**: 두 안 비교 → 페르소나 반응 비교 → 당사 정보 중심 장단점 → FP 판매·마케팅 전략
2. **비교 방식**: A/B 좌우 분할 UI
3. **당사 정보**: 매 요청마다 사용자가 직접 텍스트로 입력 (저장 없음, 1회용)
4. **입력 모드**: 3종 토글
   - (a) 상품설명서·약관 텍스트
   - (b) 마케팅 카피·광고 문구
   - (c) 상품 컨셉 + 핵심 보장 요약
5. **FP 리포트 깊이**: 타겟 페르소나별 어프로치 스크립트 + 채널 추천 (대면/디지털/지역 등)

### 인터랙션 노트

- **결과 영역**: 분석 완료 후에만 노출. 그 전엔 EmptyState
- **반응형**: lg 미만에서 좌우 분할 → 세로 스택. 비교 표는 가로 스크롤 허용

## 3. API 설계

### 옵션 1: 기존 `/api/analyze` 두 번 호출 + 클라이언트에서 합치기

| 항목 | 내용 |
|------|------|
| **장점** | 기존 코드 재사용 100%. 백엔드 신규 코드 0 |
| **단점** | 비교 리포트·당사 장단점·FP 전략은 어차피 새 엔드포인트 필요. 클라이언트 페이로드 큼 |

### 옵션 2: 신규 `/api/abtest` 단일 엔드포인트  ⭐ 추천

| 항목 | 내용 |
|------|------|
| **장점** | 백엔드에서 A·B 분석을 `asyncio.gather`로 병렬 처리. 클라이언트 호출 1회. 페이로드 단순 |
| **단점** | 신규 코드 작성 필요 |

**추천 근거**:
1. 비교 리포트는 어차피 새 LLM 호출 필요 → 새 엔드포인트는 어차피 필요
2. 백엔드 병렬화로 단일 분석과 비슷한 체감 시간(~22초)
3. 영속화도 한 레코드로 통일

### 엔드포인트 설계

```
POST /api/abtest
{
  company_context: string,           // 당사 정보 (300~2000자)
  input_mode: "terms" | "marketing" | "concept",
  variant_a: {
    label: string,                   // 별명 (예: "현재안")
    text: string                     // 본문 (20~20000자)
  },
  variant_b: { 동일 구조 },
  llm_provider: "anthropic" | "sllm",
  top_k?: number                     // 기본 50
}

→ Response:
{
  abtest_id: string,
  variant_a: {
    selling_points, top_personas, top_opinions, population_stats, elapsed_ms
  },
  variant_b: { 동일 구조 },
  comparison: {
    summary_table: [{ label, a_value, b_value, delta }],
    category_diff: {...}
  },
  company_insights_md: string,
  fp_strategy_md: string,
  recommended_variant: "A"|"B"|"split",
  elapsed_ms: { extract_a, extract_b, score_a, score_b, opinions_a, opinions_b, compare, strategy, total }
}
```

### LLM 호출 수와 비용 통제

| 단계 | 호출 수 | 비고 |
|------|---------|------|
| 소구점 추출 A | 1 | 병렬 |
| 소구점 추출 B | 1 | 병렬 |
| 의견 생성 A | 10명 (단일 분석 20명 → A/B는 10명씩 축소) | |
| 의견 생성 B | 10명 | |
| 비교 리포트 1 (당사 장단점) | 1 | 신규 프롬프트 `abtest_company.md` |
| 비교 리포트 2 (FP 전략) | 1 | 신규 프롬프트 `abtest_strategy.md` |
| **합계** | **약 24** | 단일 분석(22)의 1.1배 |

**의견 축소**: 단일은 top 20 + bottom 20 = 40콜, A/B는 top 10씩 = 20콜. bottom 생략(MVP 불필요).

## 4. 결과 리포트 구조와 데이터 소스

| 섹션 | 데이터 소스 | 생성 주체 |
|------|------------|----------|
| **[1] 한눈에 비교 카드** | A·B `population_stats`, 평균 score, `persona_category_weights` | 클라이언트 계산 |
| **[2] 좌우 분할 결과** | A·B 각각 `selling_points`, `top_personas[:5]`, `top_opinions[:3]`, `province_stats[:5]` | `/api/analyze` 로직 재사용 |
| **[3] A vs B 비교 표** | 양쪽 핵심 지표 | 백엔드 `services/comparison.py` (LLM 미사용) |
| **[4] 당사 관점 장단점** | `company_context` + A/B 소구점 + 비교 표 요약 | LLM 1회 (`abtest_company.md`) |
| **[5] FP 판매전략** | `company_context` + 비교 표 + 핵심 타겟 인구통계 | LLM 1회 (`abtest_strategy.md`) |

### 비교 표 항목 (예시 9~10개)

| 항목 | A | B | 차이 |
|------|---|---|------|
| 평균 반응도 점수 | 67.3 | 71.8 | +4.5 (B 우위) |
| 타겟층 규모 (상위 5%) | 48,200명 | 52,100명 | +3,900명 |
| 핵심 타겟 규모 (상위 0.5%) | 4,820명 | 5,210명 | +390명 |
| 1순위 페르소나 카테고리 | family 0.55 | professional 0.45 | 분기 |
| 평균 가입의향 (의견 샘플) | 3.2 / 5 | 3.8 / 5 | +0.6 |
| 긍정 의견 비율 | 60% | 78% | +18%p |
| 1순위 시도 | 경기 (29%) | 서울 (33%) | 분기 |
| 우세 연령대 | 40대 | 30대 | 분기 |
| 우세 가구 유형 | 배우자+자녀 | 1인가구 | 분기 |

> 항목 확정은 구현 단계에서 추가 합의

### FP 판매전략 마크다운 구조 (프롬프트 가이드)

```
## 추천안
(A 또는 B 또는 "타겟별 분기 운영" — 근거 1~2문장)

## 타겟별 어프로치 스크립트
### 타겟 1: (인구통계 한 줄)
- **추천안**: A or B
- **오프닝 한 문장**: "..."
- **핵심 후킹 포인트**: ...
- **예상 반론과 응대**: ...

### 타겟 2, 3: 동일 구조

## 채널 추천
- **대면(설계사)**: 어느 타겟에 적합한지 + 근거
- **디지털(모바일·웹)**: ...
- **지역 거점**: 1순위 시도/시군구 명시
- **제휴 채널**: (해당되면)
```

## 6. 단계 구분 (Phase)

### Phase 1: 스켈레톤 (빈 페이지 동작)

- **검증**: 네비 → 페이지 진입, `curl /api/abtest` → 200 OK

### Phase 2: 분석 로직 (A/B 병렬 분석 + 비교 표)

### Phase 3: 리포트·전략 출력 (LLM 2회 추가)
- 신규 프롬프트 2개
- `services/llm.py`에 2개 함수 추가
- `routes/abtest.py`에서 호출 추가
- `ABTestResultPanel`에 마크다운 섹션 + 추천안 배지
- **검증**: 두 마크다운 리포트 정상 출력, 추천안 표시, 한 사이클 30~60초

각 Phase 끝마다 사용자에게 보여줄 수 있는 상태 보장.

## 7. 리스크 / 미정 사항

### 리스크

| 리스크 | 가능성 | 영향 | 대응 |
|--------|--------|------|------|
| LLM 호출 비용·시간 증가 | 중 | 중 | 의견 10명씩 축소. provider 토글 |
| 당사 정보 토큰 초과 | 중 | 높 | 2000자 hard cap, 분석 시작 비활성 |
| A·B 본문 토큰 초과 | 낮 | 중 | 기존 `MAX_PRODUCT_TEXT_CHARS=8000` truncate 적용 |
| A·B 거의 동일해 비교 의미 없음 | 중 | 낮 | "차이 미미" 안내 배너 표시 |
| 응답 30초 초과 → proxy ECONNRESET | 중 | 높 | A·B 병렬화, opinion 축소, Next.js 타임아웃 확인 |
| 추천안 LLM이 명확히 한쪽 못 고름 | 중 | 낮 | "타겟별 분기" 선택지, 프롬프트 명시 |
| 입력 모드 토글이 로직에 영향 안 줌 | 낮 | 낮 | MVP는 메타로만 (프롬프트 톤 힌트로 사용) |

### 미정 — 대표님 의사결정 필요

1. **이력 저장 여부**: A/B 결과를 `data/abtests.jsonl`에 영속화? MVP는 저장 없음 가능하지만 향후 이력 탭 추가 고려. **추천: 영속화 ON, UI는 v2**
2. **비교 표 항목 확정**: 위 9개 항목 가감
3. **별명 기본값**: "안 A"/"안 B" vs "현재안"/"리뉴얼안"
4. **입력 모드별 라벨/플레이스홀더 톤**
5. **추천안 표시 강도**: 약한 쪽 시각적 약화 적용 여부
6. **상태 관리**: 이탈 시 결과 보존 여부. MVP는 페이지 state로만
7. **`/api/simulate` 연동**: "이 페르소나에 설문 돌리기" 버튼. **추천: v2 후보**

## 8. 검증 계획 (MVP 완료 시점)

## 9. 의존성 / 호환성

- **기존 API 영향 없음**: `/api/analyze`, `/api/simulate` 그대로
- **기존 UI 영향 없음**: `/`, `/surveys`, `/personas`, `/overview` 그대로
- **공유 자산 재사용**:
  - 백엔드: `extract_selling_points`, `embed_text`, `score_personas`, `generate_persona_opinions`, `get_store()`
  - 프론트: `LLMProviderToggle`, `AnalysisProgress`, `ReportPanel`, 한화 톤 토큰
- **의견 N 축소가 단일 분석에는 영향 없음**: A/B 라우트만 자체 10명. 기존 분석은 변경 없음
